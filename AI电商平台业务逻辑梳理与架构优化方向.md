# AI 电商平台业务逻辑梳理与架构优化方向

> 整理日期：2026-07-03  
> 适用范围：AI 电商智能体平台现有图片创作、复刻、模块、模板、广场、创作记录与新增扩图能力  
> 文档目的：在没有口头交接的情况下，先站在电商客户业务视角理解平台逻辑、当前问题与后续优化方向，为后续产品架构重构和扩图功能设计提供依据。

## 1. 平台业务定位

当前平台不应被理解为单一的 AI 生图工具，而应理解为面向电商客户的“商品视觉内容生产与资产复用平台”。

电商客户的目标不是单次生成一张图片，而是围绕商品完成持续的视觉内容生产：

1. 准备商品素材。
2. 生成商品主图、模特图、详情页套图、营销图。
3. 对生成结果进行二次处理，如加字、扩图、修复、换比例。
4. 将满意结果保存为成品或沉淀为可复用模块。
5. 用模块和模板批量复用，提升后续出图效率。
6. 通过创作记录追溯任务，通过资产库管理结果，通过广场购买成熟资产。

因此，平台核心价值应从“AI 能生成图片”升级为：

- 降低电商视觉内容制作门槛。
- 让商品图、模特、模块、模板、文字样式等资产可沉淀、可复用、可交易。
- 将一次创作结果转化为后续批量生产能力。
- 支持电商主图、详情页、海报、方图、竖图等多场景交付。

## 2. 核心业务对象

现有资料中已经隐含了多个核心对象，需要在产品架构中明确区分。

| 业务对象 | 定义 | 主要用途 |
| --- | --- | --- |
| 商品素材 | 商品图、颜色、类目、卖点、材质、细节图等 | 所有创作的基础输入 |
| 模特资产 | AI 生成或上传处理后的可复用模特 | 用于模特图、服饰展示、视频或图片创作 |
| 生成任务 | 一次自由生图、复刻、模块生图、模板生图、扩图等操作 | 记录过程、状态、参数和结果 |
| 生成结果 | 某次任务输出的图片、套图或长图 | 下载、继续编辑、加入资产 |
| 模块 | 单个可复用创作单元，包含素材约束、提示词、场景描述、效果图 | 后续基于模块快速生成同类图片 |
| 模板 | 多个模块按顺序组成的详情页或套图结构 | 批量生成详情页套图 |
| 文字样式 | 可复用的文字、水印、排版样式 | 给图片继续加字或做营销视觉 |
| 成品图片 | 已完成、可下载、可商用、可继续编辑的最终图 | 作为客户真正交付物 |
| 广场资产 | 被发布到公域售卖的模块、模板、模特、文字样式 | 购买、复用、交易 |
| 购买记录 | 用户在广场购买资产产生的订单记录 | 查订单、查来源、回到资产 |
| 创作记录 | 平台内所有创作任务的过程记录 | 返回任务、查看状态、追溯失败原因 |

关键边界：

- 创作记录不是资产库。它记录“做过什么任务”。
- 成品库或图片资产库才记录“可直接使用的最终结果”。
- 模块和模板不是普通图片结果，而是可复用的生产规则。
- 广场不是创作入口，而是资产交易与复用入口。

## 3. 电商客户的真实业务路径

从客户视角看，平台应围绕“我要给商品做图”组织路径，而不是围绕技术能力平铺。

### 3.1 单张商品图生产路径

1. 用户上传商品图或从商品素材库选择商品。
2. 用户输入图片需求，例如主图、场景图、模特图、白底图、详情图。
3. 用户可上传参考图、模特图、背景图、包装图等辅助素材。
4. 用户选择比例、分辨率、模型。
5. 系统生成图片。
6. 用户在结果图上继续操作：
   - 下载。
   - 重新生成。
   - 加文字。
   - 扩图。
   - 局部修复。
   - 加入成品库。
   - 加入个人模块。
7. 任务进入创作记录，结果进入成品或资产体系。

### 3.2 参考图复刻路径

1. 用户上传一张参考图。
2. 系统提取参考图的构图、背景、光影、商品表现、风格。
3. 用户上传自己的商品图。
4. 系统生成类似风格或类似构图的新图。
5. 用户可继续扩图、加字、下载、保存成品。

### 3.3 模块复用路径

1. 用户先通过一次成功创作沉淀模块，或从模块广场购买模块。
2. 用户进入基于模块生图。
3. 系统根据模块约束要求用户上传对应类型和数量的素材。
4. 自创模块允许用户看到并修改提示词，购买模块不展示提示词。
5. 系统按模块规则生成新图。
6. 用户下载、加字、扩图或继续保存结果。

### 3.4 模板套图路径

1. 用户创建模板，或从模板广场购买模板。
2. 模板由多个模块组成，对应详情页结构。
3. 用户进入基于模板生图。
4. 系统按模块顺序要求用户补齐素材。
5. 图片创作模块调用 AI 生成，固定图片模块直接展示，文字模块加载文字 JSON。
6. 生成后形成多张结果图和长图预览。
7. 用户可勾选、排序、自定义添加图片、下载套图、继续添加文字。

### 3.5 任务回流路径

1. 用户进入创作记录。
2. 按创作类型、名称、创作人、时间、状态筛选。
3. 用户点击返回任务。
4. 系统回填当时的素材、参数、提示词、结果和任务状态。
5. 用户继续查看、下载、重新生成或二次编辑。

## 4. 当前产品架构存在的问题

### 4.1 菜单按功能平铺，缺少业务主线

当前菜单存在大量并列功能：

- 自由生图。
- 基于模块生图。
- 基于模板批量生图。
- AI 模特创建。
- 文字/水印创建。
- 参考图复刻。
- 套图批量复刻。
- 模块广场。
- 模板广场。
- 模特广场。
- 创作记录。

这些能力本身有价值，但对电商客户来说，入口过散。

客户真正关心的是：

- 我要做商品主图。
- 我要做详情页套图。
- 我要复刻参考图。
- 我要把生成图改成平台需要的尺寸。
- 我要给图加文字。
- 我要找历史结果继续处理。

现有菜单没有把这些目标路径聚合起来，导致用户需要先理解平台内部能力，再反推自己该点哪个入口。

### 4.2 创作、资产、记录、广场边界混在一起

目前多个概念容易混淆：

- 创作记录里展示的是任务，但用户可能误以为是图片资产。
- 基于模块生图既像资产列表，又像创作入口。
- 广场和个人资产之间的关系没有被显式表达。
- 成品图片与过程任务没有清晰分层。

建议明确：

- 创作页：负责发起任务。
- 任务中心：负责查看过程和状态。
- 成品中心：负责管理最终可用图片。
- 资产中心：负责管理模块、模板、模特、文字样式。
- 广场中心：负责购买和发布资产。

### 4.3 “结果后二次处理”能力没有形成统一链路

电商图片生成后，用户常见的下一步不是结束，而是继续加工：

- 加文字。
- 扩图。
- 调整比例。
- 下载不同尺寸。
- 修复局部问题。
- 保存成品。
- 替换到套图。

当前这些动作分散在不同页面或尚未形成统一结果操作区。扩图功能如果单独做成入口，会继续加重这种分散。

更合理的方式是：所有结果图下方统一提供二次处理动作。

### 4.4 资产复用价值没有在入口层表达清楚

模块和模板是平台区别于普通 AI 生图工具的关键能力。

但如果用户不理解：

- 为什么要加入个人模块。
- 模块和模板有什么区别。
- 自创模块和购买模块有什么权限差异。
- 模块如何进入模板。
- 模板如何生成详情页套图。

那么这些功能就会显得复杂。

需要在信息架构中强化：

- 模块 = 单张图创作规则。
- 模板 = 多个模块组成的一套详情页方案。
- 广场 = 购买别人沉淀好的模块/模板。
- 资产中心 = 管理自己拥有的模块/模板/模特/文字样式。

### 4.5 现有比例和尺寸能力不足以覆盖电商交付

电商客户经常需要多平台尺寸：

- 主图方图。
- 详情页竖图。
- 海报竖图。
- 横版活动图。
- 小红书/抖音等内容图。
- 平台指定像素，如 1200x1200。

单纯选择生图比例无法解决后续尺寸适配，所以扩图能力应作为结果处理链路的一部分补齐。

## 5. 推荐的产品架构优化方向

建议参考 AI 视频 Demo 的思路：将平台从“功能菜单集合”重构为“业务工作台 + 资源中心 + 任务中心 + 资产中心”。

### 5.1 一级架构建议

| 一级模块 | 定位 | 包含内容 |
| --- | --- | --- |
| 创作工作台 | 用户主要生产入口 | 自由生图、参考图复刻、主图/详情图/套图创作、结果二次处理 |
| 商品素材库 | 商品基础资料管理 | 商品图、颜色、类目、卖点、材质、细节图 |
| 资源中心 | 可复用创作资源 | 模特库、模块库、模板库、文字样式库 |
| 成品中心 | 最终可用图片管理 | 已生成图片、扩图结果、加字结果、套图结果 |
| 任务中心 | 过程记录和回流 | 生成中、已完成、失败、草稿、返回任务 |
| 广场中心 | 资产交易与复用 | 模块广场、模板广场、模特广场、文字样式广场 |
| 账户与算力 | 交易与权限 | 购买记录、融豆消耗、套餐、账户信息 |

### 5.2 创作工作台建议

创作工作台应成为主入口，承接用户“我要做图”的需求。

可以包含以下创作目标：

- 自由生图。
- 参考图复刻。
- 商品主图。
- 模特展示图。
- 详情页套图。
- 白底图/细节图。

用户完成生成后，所有结果图统一进入结果操作区：

- 下载。
- 重新生成。
- 扩图。
- 添加文字。
- 局部修复。
- 保存成品。
- 加入模块。
- 替换到套图。

### 5.3 资源中心建议

资源中心管理可复用资产，不直接承接一次性生图任务。

建议拆分：

- 商品素材库。
- 模特库。
- 模块库。
- 模板库。
- 文字样式库。

每类资源都需要区分：

- 自创。
- 已购。
- 已发布。
- 草稿。

### 5.4 任务中心建议

任务中心承接所有过程记录。

任务类型建议包含：

- 自由生图。
- 参考图复刻。
- 模块生图。
- 模板生图。
- 模特创建。
- 文字编辑。
- 扩图。
- 修复。

任务中心的核心操作是：

- 查看详情。
- 返回任务。
- 查看失败原因。
- 重试。
- 下载结果。

### 5.5 成品中心建议

成品中心应专门管理客户真正可用的交付结果。

建议记录：

- 原始生成图。
- 扩图结果。
- 加字结果。
- 局部修复结果。
- 套图结果。
- 长图预览。

成品中心支持：

- 按商品、类目、时间、任务类型筛选。
- 下载。
- 继续编辑。
- 继续扩图。
- 加入模板或模块。
- 查看来源任务。

## 6. 扩图功能在整体业务中的定位

扩图不建议作为一级菜单或独立主入口优先出现。

扩图应定位为“生成结果后的二次处理能力”，属于图片成品处理链路。

### 6.1 扩图的入口位置

扩图入口应出现在所有成功生成图片的操作区中：

- 自由生图结果卡片。
- 参考图复刻结果卡片。
- 基于模块生图结果卡片。
- 基于模板生图中的单张结果。
- 创作记录返回任务后的结果图。
- 成品中心的图片详情页。

推荐按钮层级：

- 查看大图。
- 下载。
- 扩图。
- 添加文字。
- 重新生成。

### 6.2 扩图解决的业务问题

扩图解决的是电商交付尺寸适配问题。

用户已有一张可用图，但需要适配不同场景：

- 方图主图。
- 竖版海报。
- 横版活动图。
- 平台指定尺寸。
- 多渠道投放尺寸。

扩图不是简单拉伸图片，而是在保持商品和模特主体不变的前提下，补全背景和画面边缘。

### 6.3 扩图的核心规则

客户已明确要求：

- 无论选择什么比例或尺寸，模特必须在画面居中展示。
- 商品主体需要完整展示。
- 服饰、蕾丝、网纱、纹理、颜色、版型等细节不能被破坏。
- 扩展区域主要补背景和场景，不应重绘坏人物和商品主体。

扩图默认规则应为：

1. 模特主体画面居中。
2. 商品区域优先保护。
3. 人物完整性优先于自由构图。
4. 扩图区域自然补全背景。
5. 输出尺寸准确。

### 6.4 扩图参数建议

扩图参数可以分为常用和高级。

常用参数：

- 原始比例。
- 1:1。
- 3:4。
- 4:5。
- 9:16。
- 16:9。

高级参数：

- 自定义宽高，如 1200x1200。
- 缩放倍率：1x、1.5x、2x。
- 输出分辨率。

不建议在 MVP 中开放过大的 3x、4x 扩展，因为会明显增加画面匹配难度和质量风险。

### 6.5 扩图记录存放建议

扩图应产生新的任务记录，同时关联原图来源。

建议记录字段：

| 字段 | 说明 |
| --- | --- |
| 任务类型 | 扩图 |
| 来源任务 ID | 原始图片来自哪个生成任务 |
| 来源图片 ID | 被扩图的图片 |
| 输出图片 | 扩图结果 |
| 目标比例 | 如 1:1、4:5、16:9 |
| 目标尺寸 | 如 1200x1200 |
| 缩放倍率 | 如 1.5x |
| 主体居中 | 默认是 |
| 状态 | 生成中、已完成、失败 |
| 创建时间 | 扩图任务时间 |

展示位置建议：

1. 原任务结果卡片下展示扩图子记录。
2. 任务中心中作为独立“扩图”任务展示。
3. 扩图结果进入成品中心。

这样既能追溯来源，也能统一管理结果。

## 7. 对后续设计的建议

在进入界面设计前，建议先完成以下产品定义：

1. 重新定义平台一级信息架构。
2. 明确任务、成品、资产、广场、购买记录的边界。
3. 统一所有生成结果的操作区。
4. 明确扩图属于结果后二次处理，不是独立创作入口。
5. 梳理成品中心是否需要承接扩图、加字、修复等二次编辑结果。
6. 梳理任务中心是否统一承接所有生成、扩图、修复、文字编辑记录。
7. 明确模块、模板、模特、文字样式的自创/已购/发布状态。

## 8. 阶段性结论

当前平台已经具备较完整的能力雏形，但产品组织方式仍偏“功能堆叠”。后续优化重点不应只是新增扩图按钮，而应围绕电商客户真实业务链路重构：

- 先让用户知道“我要做什么图”。
- 再让用户选择素材和创作方式。
- 生成后统一提供二次处理能力。
- 结果沉淀到成品中心。
- 好的创作沉淀为模块和模板。
- 所有过程进入任务中心可追溯。
- 可复用资产进入资源中心或广场流通。

扩图功能在这个体系中应作为“结果图尺寸适配与场景延展能力”，服务于电商多尺寸交付，而不是成为一个孤立的新模块。

## 9. AI 视频 Demo UI 规范与 AI 电商改版基准

本阶段先暂停具体页面设计，先将 AI 视频 HTML Demo 的视觉和交互规范沉淀为 AI 电商改版的设计基准。后续 AI 电商的菜单骨架、商品库、创作工作台、资产中心、任务记录等页面设计，都应优先沿用这套基准。

### 9.1 整体风格

AI 视频 Demo 的整体风格适合作为 AI 电商改版方向：

- 企业级 AI 工作台。
- 浅蓝灰页面背景。
- 白色内容卡片。
- 蓝紫色主色。
- 轻量卡片。
- 克制圆角。
- 强调任务流、资源选择、状态反馈和结果回流。

AI 电商改版不应走营销落地页风格，也不应做成装饰性很强的图片工具页面，而应保持工作台式、清晰、稳定、可重复操作的产品气质。

### 9.2 色彩规范

| 用途 | 颜色 |
| --- | --- |
| 主色 | `#505cff` |
| 主色浅底 | `#eef1ff` |
| 页面背景 | `#f6f8fe` / `#f1f4ff` |
| 卡片背景 | `#ffffff` |
| 主文字 | `#172033` |
| 次级文字 | `#657084` |
| 弱文字 | `#98a2b3` |
| 描边 | `#e6eaf2` |
| 成功 | `#12b76a` |
| 警告 | `#f79009` |
| 失败 | `#f04438` |

使用原则：

- 主色用于关键按钮、选中态、重点操作。
- 主色浅底用于菜单选中、轻量标签、弱强调背景。
- 页面背景保持浅蓝灰，承托白色内容区。
- 状态色只用于任务状态、校验反馈、风险提示，不用于大面积装饰。

### 9.3 布局规范

页面骨架沿用 AI 视频 Demo：

- 顶部栏 + 左侧菜单 + 右侧主工作区。
- 顶部栏高度约 `64px`。
- 左侧菜单宽度约 `224px`。
- 右侧主工作区承载页面核心内容。
- 页面主体使用浅蓝灰背景，内容模块使用白色卡片承载。

布局原则：

- 主路径页面优先使用工作台布局。
- 表单、筛选、列表、结果区之间保持清晰分区。
- 不在一个页面内堆叠过多无关功能。
- 高频操作应靠近用户当前上下文，如生成结果图下方展示下载、扩图、加字等操作。

### 9.4 菜单规范

菜单采用“图标 + 文案”的形式。

选中态：

- 浅蓝紫底。
- 主色文字。
- 图标同步高亮。

菜单组织原则：

- 一级菜单代表业务域。
- 二级菜单代表业务模块。
- 三级菜单代表具体功能或资源分类。
- 避免把一次性工具、资产管理、任务记录混在同一层级。

### 9.5 卡片规范

卡片用于列表项、资源项、任务项、弹窗内容块等。

基础样式：

- 白色背景。
- 轻描边。
- `8px` 或 `12px` 圆角。
- Hover 时轻阴影。

使用原则：

- 商品、模块、模板、模特、成品图等可用卡片展示。
- 卡片外部只展示决策所需摘要信息。
- 复杂详情进入详情页或右侧抽屉，不堆在卡片上。
- 卡片操作应克制，优先展示主操作，其他操作放更多菜单或 hover 层。

### 9.6 按钮规范

按钮层级：

| 类型 | 用法 |
| --- | --- |
| 主按钮 | 生成、确认、保存、提交等关键动作 |
| 次按钮 | 取消、返回、筛选、普通编辑 |
| 图标按钮 | 下载、预览、更多、删除、放大查看 |

样式建议：

- 主按钮使用蓝紫渐变或主色填充。
- 次按钮使用白底描边。
- 图标按钮用于轻量操作，避免所有动作都做成大文字按钮。
- 危险操作使用红色，但需要二次确认。

### 9.7 表单规范

表单适用于商品信息、创作参数、素材上传、模块保存、模板配置等场景。

基础样式：

- 输入框浅描边。
- 聚焦态使用主色描边。
- 聚焦态可增加浅色阴影。
- 表单项应分组，不应长列表无分隔。

交互原则：

- 必填项明确标识。
- 参数选择应同步生成摘要。
- 上传后应显示缩略图和上传状态。
- 多图上传需要显示序号、删除、放大预览。

### 9.8 状态标签规范

状态标签用于任务、资产、商品完整度、购买状态等。

| 状态 | 颜色建议 |
| --- | --- |
| 已完成 / 成功 | 绿色 `#12b76a` |
| 生成中 / 执行中 | 蓝紫主色 `#505cff` |
| 失败 | 红色 `#f04438` |
| 警告 / 待补充 | 橙色 `#f79009` |
| 待处理 / 草稿 | 灰色 |

状态标签的含义必须稳定，不能同一个颜色在不同页面表达不同业务含义。

### 9.9 弹窗与抽屉规范

弹窗和抽屉用于新增、编辑、详情、确认、复杂配置。

基础样式：

- 白色背景。
- `16px` 圆角。
- 轻阴影。
- 标题、说明、内容、底部操作分区清晰。

使用建议：

- 新增/编辑类内容可用弹窗。
- 商品详情、资产详情这类信息较多的内容，优先使用右侧抽屉或独立详情页。
- 删除、购买、保存等风险或关键动作使用确认弹窗。

## 10. AI 电商改版方向

### 10.1 视觉与骨架方向

AI 电商改版应沿用 AI 视频 Demo 的视觉风格和页面骨架：

- 视觉风格沿用 AI 视频。
- 页面骨架沿用 AI 视频。
- 菜单层级按 AI 电商业务重新搭建。
- 工作台式布局优先。
- 信息结构优先于装饰风格。

### 10.2 商品信息改为商品库

现有“商品信息”建议改名为“商品库”。

原因：

- “商品信息”更像一个字段页或资料页。
- “商品库”更符合电商客户理解，表示可被创作工具调用的商品素材集合。
- 后续商品图、颜色、货号、品牌、类目、卖点、模特图、AI 生成记录等都应围绕商品库沉淀。

商品库可参考 AI 视频 Demo 的商品素材库布局，但 AI 电商商品字段更多，需要增强筛选、搜索和详情能力。

### 10.3 商品卡片信息边界

商品卡片外部只展示识别和筛选所需的摘要信息：

- 商品图。
- 商品名称。
- 品牌。
- 商品唯一码。
- 货号。
- 品类。
- 状态。
- 更新时间。

不建议在卡片上直接堆叠所有复杂信息。

以下信息应进入商品详情：

- 卖点。
- 材质。
- 尺码。
- 颜色。
- 价格。
- 模特图。
- AI 生成记录。
- 商品多视角图片。
- 商品细节图。
- 商品适用场景。
- 商品历史创作结果。

商品详情建议使用右侧抽屉或独立详情页，不建议全部堆在卡片上。

### 10.4 资产中心、任务记录、商品库拆分原则

商品库、资产中心、任务记录需要拆清楚，不要重复承载同一类内容。

拆分原则：

- 商品库：管理商品基础资料和商品素材。
- 资产中心：只放可复用资产和可交付成品。
- 任务记录：只放生成过程、执行状态、失败原因和任务结果追溯。

更具体地说：

- 商品库关注“这个商品是什么”。
- 任务记录关注“我对它做过什么任务”。
- 资产中心关注“我沉淀了什么可复用或可交付结果”。

### 10.5 资产中心的边界

资产中心只放可复用资产，不应混入所有过程任务。

资产中心建议包含：

- 成品图片。
- 模块资产。
- 模板资产。
- 模特资产。
- 文字样式。

例如：

- AI 模特创建任务完成后，任务记录中保存任务过程；用户确认保存后，模特才进入模特资产。
- 图生图模块创作完成后，任务记录中保存生成过程；用户点击加入模块后，才进入模块资产。
- 扩图任务完成后，任务记录中保存扩图过程；扩图结果作为成品图片进入资产中心。

## 11. AI 电商建议菜单

### 11.1 一级菜单

一级菜单仍然保持业务域划分：

- AI 电商智能体。
- AI 营销智能体。
- AI 客服智能体。

### 11.2 AI 电商智能体二级菜单

AI 电商智能体下建议二级菜单：

- 商品库。
- 创作工作台。
- 资产中心。
- 任务记录。
- AI 广场。
- 算力与账单。
- 操作说明。

### 11.3 AI 电商智能体三级菜单

#### 商品库

- 商品素材。
- 商品分组/类目管理。

#### 创作工作台

- 自由生图。
- 参考图复刻。
- 基于模块生图。
- 基于模板生图。
- 扩图编辑。
- 文字/水印编辑。

#### 资产中心

- 成品图片。
- 模块资产。
- 模板资产。
- 模特资产。
- 文字样式。

#### 任务记录

- 全部任务。
- 生图任务。
- 复刻任务。
- 扩图任务。
- 模块/模板任务。
- 失败任务。

#### AI 广场

- 模块广场。
- 模板广场。
- 模特广场。
- 文字样式广场。

#### 算力与账单

- 购买算力。
- 账单明细。
- 购买记录。

### 11.4 菜单调整原则

- 不再把企业资产放在个人中心下作为隐藏资产仓库。
- 商品库、资产中心、任务记录应成为清晰独立的业务模块。
- 创作工作台负责发起创作和二次编辑。
- 任务记录负责过程追溯。
- 资产中心负责结果沉淀和复用。
- AI 广场负责购买和发布可复用资产。

## 12. 下一步建议

下一步建议按以下顺序推进：

1. 基于本规范确认 AI 电商一级、二级、三级菜单骨架。
2. 先设计 AI 电商整体菜单骨架，不进入细页面。
3. 以“商品库”作为第一个落地页面，参考 AI 视频 Demo 的商品素材库，但增强电商商品字段与详情能力。
4. 明确商品库、资产中心、任务记录之间的数据流转关系。
5. 再进入商品库 HTML 页面设计。

在开始页面设计前，必须先确认：

- 商品库只管理商品基础资料和素材。
- 资产中心只管理可复用资产和成品。
- 任务记录只管理任务过程和执行结果。
- 扩图编辑属于创作工作台中的二次编辑能力，同时扩图任务进入任务记录，扩图结果进入成品图片。
